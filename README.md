## Задание 1

### Решение

Для того, чтобы предложить более подходящие решения, для начала необходимо проанализировать текущее решение и его проблемы.

Исходя из условия задания, можно явно выделить следующие ключевые проблемы:

1. Нарушение принципа единственности ответственности (SRP) - в классе CourseManager, который взял на себя слишком много обязанностей, содержится и бизнес-логика, создание объектов, отправка уведомлений и интеграции с внешними сервисами.

2. Нарушения принципа открытости/закрытости (OCP) - при добавлении нового типа курса необходимо изменять существующий код. К этому же относится создание курсов через прямые вызовы конструкторов. А также жёсткая фиксация алгоритма расчёта оценки в одном методе.

3. Нарушен принцип инверсии зависимостей (DIP) - основной класс зависит от конкретных API интеграций, конкретных классов внешнего API, вызывает их напрямую.

4. Нарушение принципа DRY (Don’t Repeat Yourself) - похожие действия выполняются в разных методах CourseManager.

Хочу обратить внимание, что из списка выше одна и та же проблема может требовать сразу несколько решений в разных местах кода. Например, нарушение OCP при добавлении нового типа курса может быть решена с помощью паттерна *FactoryMethod*, а жёсткая фиксация алгоритма этого же принципа - с помощью *Strategy*.

## Задание 2

### Решение

Прежде чем предлагать изменения, необходимо разобрать, почему система рушится именно при пиковой нагрузке, хотя суммарный объём запросов за сутки сервис ресторанов теоретически способен обработать.

Из условия следует цепочка отказов: мобильное приложение направляет весь поток на первый экземпляр сервиса бронирования. Тот синхронно вызывает единственный экземпляр сервиса ресторанов. При перегрузке падает и бронирование, и рестораны. При этом бизнес допускает асинхронную обработку заявки в течение 24 часов, а масштабировать можно только сервис бронирования и вспомогательные компоненты.

Ниже выделены две центральные архитектурные проблемы и способы их устранения.

#### Проблема 1. Синхронная связанность сервисов и отсутствие сглаживания пиковой нагрузки

**Суть проблемы.** Каждый запрос пользователя немедленно порождает синхронный сетевой вызов из сервиса бронирования в сервис ресторанов. Из-за этого кратковременный всплеск (до 50 000 запросов в секунду) напрямую передаётся на единственный экземпляр сервиса ресторанов, который физически не рассчитан на такую интенсивность.

При этом аналитики установили важный факт: если распределить тот же суточный объём равномерно, сервис ресторанов использует лишь около 20% своей пропускной способности. Значит, узкое место - не суммарная нагрузка, а её концентрация во времени. Синхронная схема "запрос пользователя = немедленная проверка и бронь в ресторане" лишает систему возможности разнести обработку по времени, хотя бизнес-требование это допускает (обработка в течение 24 часов).

**Способ устранения.** Ввести асинхронную обработку через промежуточную очередь сообщений, разместив её в ДЦ1.

Схема работы:

1. Мобильное приложение отправляет заявку на бронирование в сервис бронирования.
2. Сервис бронирования проверяет запрос, сохраняет заявку со статусом "принята" и помещает её в очередь. Пользователю сразу возвращается подтверждение приёма заявки, а не результат бронирования.
3. Отдельный фоновый обработчик (или несколько обработчиков) с ограниченной скоростью читает сообщения из очереди и вызывает сервис ресторанов с контролируемой интенсивностью, такой, чтобы нагрузка на единственный экземпляр оставалась примерно равномерной.
4. После успешного бронирования обработчик обновляет статус заявки и уведомляет пользователя через приложение.

Таким образом, пиковый поток распределяется по временному окну до 24 часов. Сервис ресторанов получает не 50 000 одновременных запросов, а стабильный поток, укладывающийся в его пропускную способность. При этом отказы на этапе приёма заявки не требуются: пользователь всегда получает подтверждение, что заявка принята в обработку.

#### Проблема 2. Неравномерное распределение запросов на стороне клиента

**Суть проблемы.** Мобильное приложение обращается к экземплярам сервиса бронирования строго по фиксированному списку адресов: сначала к первому, и только при его недоступности - ко второму, затем к третьему и так далее. При пиковой нагрузке это приводит к эффекту "стада": практически все 50 000 запросов в секунду в первые секунды падают на один экземпляр, который перегружается и отказывает раньше остальных.

В результате даже при наличии множества экземпляров сервиса бронирования в ДЦ1 они используются последовательно, а не параллельно. Первые экземпляры в списке получают непропорционально большую долю запросов и выходят из строя, после чего нагрузка смещается на следующий экземпляр по той же схеме. Это ускоряет каскадный отказ всей подсистемы бронирования ещё до того, как запросы дойдут до сервиса ресторанов.

**Способ устранения.** Разместить перед экземплярами сервиса бронирования балансировщик нагрузки в ДЦ1.

Схема работы:

1. В мобильном приложении вместо списка адресов всех экземпляров указывается единый адрес балансировщика.
2. Балансировщик распределяет входящие запросы между экземплярами по очереди или по наименьшей текущей загрузке, обеспечивая примерно равномерное использование всех экземпляров.
3. При отказе отдельного экземпляра балансировщик автоматически исключает его из списка и перенаправляет запросы на работающие узлы, не требуя от клиента перебора адресов.

Это позволяет задействовать все экземпляры сервиса бронирования одновременно и снизить вероятность их одновременного падения под пиковой нагрузкой. В сочетании с очередью из первой проблемы балансировщик принимает и быстро подтверждает большой поток заявок, а обращение к сервису ресторанов выполняется фоновыми обработчиками с ограниченной скоростью.

| Проблема | Решение | Эффект |
|----------|---------|--------|
| Синхронная связанность и пиковая нагрузка | Асинхронная очередь и фоновые обработчики с ограничением скорости | Нагрузка на сервис ресторанов выравнивается во времени, укладывается в 20% мощности |
| Неравномерное распределение запросов на клиенте | Балансировщик нагрузки перед экземплярами бронирования | Запросы распределяются равномерно, каскадный отказ экземпляров замедляется |

#### Вывод

В исходной архитектуре выявлены две взаимосвязанные проблемы: прямая синхронная передача пикового потока запросов на единственный экземпляр сервиса ресторанов и неэффективное распределение запросов между экземплярами сервиса бронирования на стороне клиента.

Предложенное решение - введение балансировщика нагрузки для равномерного приёма заявок и асинхронной очереди с контролируемой скоростью обработки - использует допустимое бизнесом окно в 24 часа и не требует масштабирования сервиса ресторанов. В результате система способна принять весь пиковый поток без отказов пользователям и обработать его в пределах суточного лимита.

## Задание 3

### Решение

По описанию видно, что в системе бизнес-правила и техническая часть перемешаны. Расчёт скидки, проверка льгот и обработка заявок написаны прямо в тех компонентах, которые принимают пользовательские запросы и одновременно работают с базой данных и внешними сервисами.

#### Выявленные проблемы

1. **Смешение бизнес-логики и инфраструктуры.** Компонент, который отвечает на запрос пользователя, заодно решает, кому положена льгота, и заодно знает, как устроена таблица в базе. Один модуль отвечает сразу за предметные правила, передачу данных и доступ к хранилищу. Это нарушение разделения ответственности.

2. **Неправильное направление зависимостей.** Бизнес-правила зависят от способа работы с базой данных, её схемы и внешнего сервиса, хотя должно быть наоборот. При смене инфраструктуры приходится переписывать участки, где по смыслу описаны скидки и льготы, а не только технический код.

3. **Сложное сопровождение и тестирование.** Чтобы проверить одно правило расчёта скидки, нужно поднимать всю связанную среду: приём запросов, базу данных и подключения к внешним системам. Любое изменение схемы данных или внешнего сервиса затрагивает больше кода, чем должно.

#### Подход и решение

Для такой ситуации уместна **слоистая архитектура**. Идея в том, чтобы разнести код по горизонтальным слоям, у каждого из которых своя зона ответственности. Верхний слой может обращаться к нижнему, но не наоборот.

Для описанной системы логично выделить четыре слоя:

1. **Слой представления** - компоненты, которые принимают запросы от пользователя и формируют ответы. Здесь нет расчёта скидок и обращений к базе, только преобразование данных "запрос - ответ".

2. **Бизнес-слой** - модули с предметной логикой: расчёт скидки, проверка льгот, правила обработки заявок. Этот слой не знает, как устроена база и как реализован приём запросов. Он получает и сохраняет данные через общие описания операций, а обращение к внешним системам выполняет тоже через них.

3. **Слой доступа к данным** - работа с базой данных, запросы к ней и подключение к внешним сервисам. Именно здесь сосредоточены все технические детали хранения и интеграций.

4. **Слой данных** - сама база данных и внешние системы.

Как это закрывает выявленные проблемы:

- **Смешение логики и инфраструктуры.** Компонент приёма запросов больше не считает скидки и не обращается к базе напрямую - он передаёт вызов в бизнес-слой. Работа с базой и приём запросов оказываются в разных слоях, а не в одном классе.

- **Неправильное направление зависимостей.** Бизнес-слой зависит от общих описаний операций с данными, а не от конкретного способа их реализации. Если поменять схему базы или внешний сервис, правится слой доступа к данным, бизнес-правила остаются без изменений.

- **Сложное сопровождение и тестирование.** Правила скидок и льгот можно проверять отдельно: подставить упрощённую реализацию хранилища и проверить логику без базы и приёма запросов. При изменении требований правки сосредотачиваются в бизнес-слое.

#### Вывод

Главная архитектурная ошибка исходной системы - в том, что разные обязанности сосредоточены в одних и тех же компонентах без чёткого разделения.

Слоистая архитектура решает это за счёт явного разграничения: представление, бизнес-логика и доступ к данным живут отдельно и связаны через общие описания операций. Систему проще развивать, менять инфраструктуру и проверять, не переписывая каждый раз предметные правила.

## Задание 4

### Решение

Сейчас у команды нет полноценной наблюдаемости: о проблемах узнают от пользователей, логи живут внутри контейнеров и пропадают при перезапуске, а метрики и признаки деградации нигде не собраны вместе. Из-за этого сложно понять, почему контейнеры периодически перезапускаются и почему растёт время ответа.

Ниже - инструменты из модуля курса, которые стоит добавить, и проблема, которую закрывает каждый из них.

#### Prometheus

Система сбора и хранения метрик. Приложение должно отдавать в неё показатели работы: время ответа, количество ошибок, число обрабатываемых запросов, доступность сервиса.

**Какую проблему решает:** команда перестаёт узнавать о сбоях только из обращений пользователей. Метрики фиксируют рост времени ответа и другие признаки деградации в момент их появления, а не постфактум.

#### cAdvisor

Средство сбора метрик контейнеров: потребление процессора, памяти, диска и сети на каждом узле.

**Какую проблему решает:** помогает связать перезапуски контейнеров с нехваткой ресурсов. Если контейнер падает из-за нехватки памяти или перегрузки процессора, это видно на графиках до и в момент перезапуска.

#### Promtail и Loki

Promtail - агент, который собирает логи с узлов и из контейнеров и отправляет их в Loki. Loki - централизованное хранилище логов с поиском по меткам.

**Какую проблему решает:** логи больше не хранятся только внутри контейнера. После перезапуска записи остаются в общем хранилище, и команде доступна информация для разбора инцидента. Это устраняет потерю данных, которая сейчас мешает анализу.

#### Grafana

Инструмент для построения дашбордов. Подключается к Prometheus как источнику метрик и к Loki как источнику логов.

**Какую проблему решает:** даёт единую картину состояния системы. На одном экране можно одновременно смотреть метрики приложения, нагрузку на контейнеры и соответствующие записи в логах. Разбор инцидента становится быстрее: не нужно вручную собирать данные с разных узлов.

| Инструмент | Проблема |
|------------|----------|
| Prometheus | Нет данных о деградации до жалоб пользователей |
| cAdvisor | Неясна причина перезапусков контейнеров |
| Promtail + Loki | Логи теряются при перезапуске контейнеров |
| Grafana | Нет единого представления по метрикам и логам |

#### Вывод

Для повышения наблюдаемости достаточно выстроить связку из четырёх элементов: Prometheus и cAdvisor собирают метрики приложения и контейнеров, Promtail с Loki централизуют логи, Grafana объединяет всё в одном интерфейсе. В результате команда может замечать проблемы самостоятельно, сохранять данные для анализа и быстрее разбирать инциденты.
